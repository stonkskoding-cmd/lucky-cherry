# LuckyCherry — Telegram Mini App казино

Одностраничное веб-приложение казино на демо-валюте («черешни»). Весь проект — **один файл `index.html`**: без сборки, без зависимостей, без бэкенда.

**Прод:** https://stonkskoding-cmd.github.io/lucky-cherry/
**Репозиторий:** https://github.com/stonkskoding-cmd/lucky-cherry

> ⚠️ **Это прототип на демо-валюте.** Вся логика — генератор случайных чисел, расчёт выплат, баланс — выполняется в браузере игрока и правится из консоли одной строкой. Для реальных денег это не подходит принципиально. См. раздел [Путь к реальным деньгам](#путь-к-реальным-деньгам).

## Быстрый старт

Файл самодостаточный, но **открывать его как `file://` нельзя** — Telegram SDK и часть API требуют http(s). Поднимите локальный сервер:

```bash
cd путь/к/проекту
python -m http.server 8000
# открыть http://localhost:8000/index.html
```

Проверять верстку нужно на узком экране: **360×720** — это базовый размер, под который всё считается.

## Подключение Telegram-бота

Делается вручную через [@BotFather](https://t.me/BotFather). Токен бота в коде **не нужен** — он понадобится только на бэкенде, когда появится проверка `initData`.

1. `/newbot` → имя → username (должен оканчиваться на `bot`) → получите токен. **Токен хранить как пароль.**
2. `/newapp` → выбрать бота → название, описание, картинку 640×360 → **Web App URL:**
   `https://stonkskoding-cmd.github.io/lucky-cherry/`
   → короткое имя (напр. `play`) — даст прямую ссылку `t.me/<bot>/play`
3. `/mybots` → бот → **Bot Settings → Menu Button** → тот же URL.

### Что приложение делает внутри Telegram

Код в `initTelegram()`:

- **Определяет Telegram по `platform !== 'unknown'`**, а не по наличию объекта. SDK создаёт `window.Telegram.WebApp` и в обычном браузере — проверка на существование объекта ничего не доказывает.
- `expand()` — на весь экран, плюс повтор при `viewportChanged` (часть клиентов открывается свёрнутыми).
- `disableVerticalSwipes()` (клиент 7.7+) — иначе скролл закрывает мини-апп.
- `setHeaderColor` / `setBackgroundColor` (6.1+) — под тёмную тему приложения.
- Автологин, если сессия ещё не восстановлена.

Все вызовы обёрнуты в проверку версии — на старых клиентах они бросают исключение.

## Архитектура файла

`index.html` (~5600 строк) состоит из:

| Часть | Что внутри |
|---|---|
| `<head>` | viewport (с запретом зума), подключение Telegram SDK |
| `<style>` | стили оболочки, затем стили каждой игры, заскоупленные под `#<id>Game` |
| разметка | оболочка (экраны, навигация), `#gameView` с хостами игр, онбординг, шторки |
| `<script>` #1 | **оболочка**: состояние, баланс, кошелёк, бонусы, аккаунты, мост `__SHELL__` |
| `<script>` #2–6 | **по одной игре**: Слоты, Кости, Мины, Crash, Рулетка |

Каждая игра — самостоятельная IIFE. Они **не видят друг друга** и общаются с оболочкой только через `window.__SHELL__` и свои хуки. Это специально: сломанная игра не должна ронять остальные.

Стили игр заскоуплены префиксом (`#slotsGame .sl-cell { … }`), чтобы имена классов не конфликтовали между играми.

## Оболочка и мост к играм

### `window.__SHELL__` — то, что игра вызывает у оболочки

```js
window.__SHELL__ = {
  getBalance(),              // текущий баланс
  setBalance(v),             // записать баланс (оболочка сама сохранит)
  onSpin(win, payout)        // раунд завершён: победа? и сумма выплаты
}
```

`onSpin` — единственная точка, через которую оболочка узнаёт о результате. Она ведёт статистику, ставит сохранение в очередь и **показывает анимацию выигрыша**. Поэтому новую игру не нужно отдельно подключать к празднованию победы — достаточно честно вызвать `onSpin`.

### Хуки, которые игра предоставляет оболочке

```js
window.__<GAME>_RESYNC__ = function(){ … }  // вызывается при открытии игры
window.__<GAME>_STOP__   = function(){ … }  // вызывается при выходе из игры
```

- **RESYNC** обязателен. Пока игра скрыта (`display:none`), её размеры равны нулю — замерять вёрстку можно только здесь. Также здесь игра забирает актуальный баланс из оболочки.
- **STOP** нужен, если игра что-то крутит: остановить таймеры, закрыть оверлеи, решить судьбу незавершённого раунда.

### Реестр игр

```js
const GAMES = {
  roulette: { title: 'Колесо Фортуны', live: true, host: 'rouletteGame', resync: '__ROULETTE_RESYNC__' },
  crash:    { title: 'Crash',          live: true, host: 'crashGame',    resync: '__CRASH_RESYNC__' },
  mines:    { title: 'Мины',           live: true, host: 'minesGame',    resync: '__MINES_RESYNC__' },
  dice:     { title: 'Кости',          live: true, host: 'diceGame',     resync: '__DICE_RESYNC__' },
  slots:    { title: 'Слоты',          live: true, host: 'slotsGame',    resync: '__SLOTS_RESYNC__' },
};
```

`openGame(key)` переключает активный хост (класс `.game-host.active`), `closeGame()` вызывает `STOP` у всех игр.

### Баланс — правило одной стороны

Игра **читает** баланс из оболочки и **никогда не навязывает свой**:

```js
let balance = (window.__SHELL__ && typeof window.__SHELL__.getBalance === 'function')
  ? window.__SHELL__.getBalance() : 1000;
```

Раньше каждая игра стартовала с `balance = 1000` и проталкивала это значение в оболочку при загрузке — из-за чего **сохранённый депозит затирался при каждой перезагрузке**. Если пишете новую игру, скопируйте строку выше дословно.

## Как добавить новую игру

Пять шагов. Пример — игра `poker`.

**1. Реестр** — добавить в `GAMES`:
```js
poker: { title: 'Покер', live: true, host: 'pokerGame', resync: '__POKER_RESYNC__' },
```

**2. Карточка** в `renderGameGrid()`:
```js
{ key:'poker', name:'Покер', sub:'Описание', art:'art-cards', emoji:'🃏', live:true },
```
Заглушки (`live:false`) показывают «Скоро». Следите, чтобы карточка не задвоилась — один ключ = одна карточка.

**3. Разметка** — рядом с другими хостами внутри `.game-frame`:
```html
<div id="pokerGame" class="game-host" data-stop="__POKER_STOP__">
  <div class="pk-app"> … </div>
</div>
```
Класс `game-host` обязателен — по нему работает переключение. `data-stop` связывает хук остановки.

**4. Стили** — заскоупить под `#pokerGame`, по образцу существующих:
```css
#pokerGame{ position:absolute; inset:0; overflow:hidden;
  align-items:stretch; justify-content:center; padding:8px 10px 10px; }
#pokerGame .pk-app{ width:100%; max-width:520px; margin:0 auto; height:100%;
  display:flex; flex-direction:column; gap:8px; }
#pokerGame .pk-app > *{ flex:0 0 auto; }
```

**5. Логика** — отдельный `<script>` с IIFE. Обязательный минимум:
```js
(function(){
  const host = document.getElementById('pokerGame');
  if (!host) return;                 // элемента нет — тихо выходим, не роняя страницу

  let balance = (window.__SHELL__ && typeof window.__SHELL__.getBalance === 'function')
    ? window.__SHELL__.getBalance() : 1000;

  function reportBalance(){
    if (window.__SHELL__ && typeof window.__SHELL__.setBalance === 'function') {
      window.__SHELL__.setBalance(balance);
    }
  }

  function settle(win, payout){
    if (payout) { balance += payout; reportBalance(); }
    if (window.__SHELL__ && typeof window.__SHELL__.onSpin === 'function') {
      window.__SHELL__.onSpin(win, payout);   // статистика + анимация выигрыша
    }
  }

  window.__POKER_STOP__   = function(){ /* стоп-таймеры, закрыть оверлеи */ };
  window.__POKER_RESYNC__ = function(){
    if (window.__SHELL__) balance = window.__SHELL__.getBalance();
    /* перерисовать UI, замерить вёрстку */
  };
})();
```

**Проверка после добавления:**
```bash
node -e "const fs=require('fs');const h=fs.readFileSync('index.html','utf8');
[...h.matchAll(/<script>([\s\S]*?)<\/script>/g)].map(m=>m[1]).forEach((b,i)=>{
```

## Правило вёрстки: без скролла

Всё должно помещаться на экран целиком — игрок не должен скроллить, чтобы увидеть кнопку ставки.

Приём: фиксированные ряды (шапка, кнопка, панель ставок) сохраняют свою высоту, а **главный элемент (поле, колесо, график) забирает остаток**. Остаток **замеряется**, а не берётся из `vh` — игра живёт под шапкой и никогда не равна высоте экрана.

```js
function fitLayout(){
  if (!host.clientHeight) return;               // скрытая игра меряет 0 — выходим
  const app = host.querySelector('.xx-app');
  const gap = parseFloat(getComputedStyle(app).rowGap) || 0;
  // элементы вне потока (absolute) не занимают ряд и не дают отступ
  const rows = [...app.children].filter(el => getComputedStyle(el).position !== 'absolute');
  let used = gap * Math.max(0, rows.length - 1);
  for (const el of rows) { if (el !== stage) used += el.getBoundingClientRect().height; }
  const cs = getComputedStyle(host);
  const avail = host.clientHeight - parseFloat(cs.paddingTop) - parseFloat(cs.paddingBottom);
  stage.style.height = Math.max(МИНИМУМ, Math.floor(avail - used)) + 'px';
}
```

Вызывать из `RESYNC`, а также на `resize` и `orientationchange`.

Так сделаны Рулетка (`fitGameLayout`), Crash (`fitCrashLayout`), Мины (`fitMinesLayout`). Кости и Слоты обходятся чистым флексом (`flex: 1 1 auto` на поле) — если хватает флекса, замер не нужен.

## Математика и RTP

**Главное правило: выплаты считаются симуляцией, а не на глаз.** Дважды на этом проекте интуиция ошибалась на порядок:

- Первая таблица Слотов выдавала **RTP 332%** — казино раздавало бы втрое больше, чем принимает. Каскады с растущим множителем компаундируются гораздо сильнее, чем кажется.
- Добавление скаттеров **уронило RTP с 96.8% до 81%** — они занимают ячейки нижнего ряда и никогда не платят сами, поэтому символов для комбинаций становится меньше. Увидеть это без прогона невозможно.

Целевой ориентир по проекту — **RTP ≈ 96%** (как в реальных слотах).

### Где заданы вероятности

| Игра | Константы | Метод |
|---|---|---|
| **Кости** | `EDGE = 0.04` | Коэффициенты **вычисляются** перебором всех 36 исходов, не вписаны руками |
| **Мины** | `BOMBS_BY_ROW = [1,1,1,1,2,2]`, `EDGE = 0.04` | Честные шансы ряда `cols/(cols-bombs)`, край снимается на каждом шаге |
| **Crash** | `GROWTH = 0.15`, `HOUSE_EDGE = 0.04` | Точка краша — обратная функция равномерного розыгрыша; край взят долей мгновенных крашей |
| **Слоты** | `MIN_MATCH = 6`, `SCATTER_CHANCE = 0.10`, `RTP_SCALE = 1.25` | `RTP_SCALE` **подобран симуляцией** на 300–400 тыс. спинов |

### Как пересчитывать

Если меняете веса символов, таблицу выплат, порог совпадений, цепочку множителей или шанс скаттера — **обязательно перемерьте**. Напишите скрипт, повторяющий логику раунда, прогоните сотни тысяч спинов и подберите масштаб:

```js
// каркас: повторить механику раунда, посчитать выплаченное / поставленное
const N = 300000, STAKE = 10;
let paid = 0;
for (let i = 0; i < N; i++) paid += simulateRound(STAKE, scale);
console.log('RTP: ' + ((paid / (N * STAKE)) * 100).toFixed(2) + '%');
```

Смотрите не только на RTP, но и на **ощущение**: доля выигрышных спинов и частота крупных заносов. Пример из практики: порог `MIN_MATCH = 5` давал корректный RTP, но выигрыши сыпались постоянно по ~1 черешне при ставке 10 — формально победа, по ощущению проигрыш. Порог 6 дал ~25% выигрышных спинов и занос ×10 примерно раз в 50 — играть стало интересно при том же RTP.

### Таблица выплат в интерфейсе

Считайте её **из тех же констант**, что и реальную выплату:
```js
const v6 = (s.pay * COUNT_MULT[MIN_MATCH] * RTP_SCALE).toFixed(2);
```
Так она физически не может разойтись с тем, что игра платит.

## Аккаунты и сохранение

Хранилище — `localStorage`:

| Ключ | Содержимое |
|---|---|
| `lc:accounts:v1` | все аккаунты: профиль, прогресс, соль и хеш пароля |
| `lc:session:v1` | id аккаунта текущей сессии |

Сохраняются: баланс, статистика, серия ежедневных бонусов, история операций, использованные промокоды. Запись отложенная (400 мс) плюс принудительная на `pagehide` и `visibilitychange` — мини-аппы убивают без предупреждения.

### Три способа входа

1. **Telegram** — аккаунт привязан к Telegram ID (`tg:<id>`).
2. **Ник + пароль** — локальный профиль (`user:<ник>`).
3. **Гость** — без сохранения.

### ⚠️ Вход по нику — это не защита аккаунта

Хеш лежит в том же браузере, что и данные, которые он «охраняет». Любой, у кого есть устройство, обходит его через консоль — как и сам баланс. Это **локальный профиль**, чтобы прогресс не терялся, и не более того. В интерфейсе стоит соответствующее предупреждение, в FAQ — тоже.

Что сделано правильно: пароль хешируется PBKDF2 (120 000 итераций, соль на аккаунт), в открытом виде **не хранится**. Если `crypto.subtle` недоступен — вход по нику отклоняется, отката к плейнтексту нет. Люди переиспользуют пароли, и утечка плейнтекста здесь утащила бы за собой другие сервисы.

## Грабли: типовые баги проекта

Это самый ценный раздел. Один и тот же класс ошибок ронял игру трижды.

### 1. `null` из декоративного элемента вешает всю панель

**Симптом:** игра работает один раз, потом панель ставок мертва до перезагрузки.

**Механика:** функция завершения раунда обращается к элементу, которого нет в разметке → исключение → выполнение обрывается на середине → флаг `spinning` навсегда остаётся `true` → все контролы заблокированы.

Реальные случаи:
- `pulseBackground()` дёргал `#ambientBg`, который потерялся при переносе рулетки из отдельной страницы (CSS переехал, разметка — нет).
- `celebrateWin()` пульсировал пилюлей баланса, которую я же удалил.

**Правила:**
- Декоративный код **обязан** переживать отсутствие элемента: `if (!el) return;`
- Раунд должен иметь **один выход** (в рулетке это `settleRound()`), чтобы новая ветка не могла вернуться раньше времени с залипшим флагом.
- Вызовы мини-игр обёрнуты так, что выплата случится **ровно один раз**: при исключении — откат к базовому множителю, плюс сторожевой таймер на случай, если колбэк не придёт.

**Проверка на битые id:**
```bash
node -e "const fs=require('fs');const h=fs.readFileSync('index.html','utf8');
const ids=[...new Set([...h.matchAll(/getElementById\(['\"]([^'\"]+)['\"]\)/g)].map(m=>m[1]))];
console.log(ids.filter(id=>!new RegExp('id=[\"\\']'+id+'[\"\\']').test(h)).join(', ')||'нет');"
```
(`balanceAmt` в выводе — известное исключение, обращение к нему закрыто проверкой.)

### 2. `requestAnimationFrame` замирает в фоне

`rAF` **перестаёт срабатывать**, когда мини-апп уходит в фон: звонок, переключение чата, гаснет экран. Если завершение раунда висит на кадрах — раунд не закроется никогда.

**Правило: исход раунда решается заранее, анимация лишь ведёт к нему, а завершение висит на `setTimeout`.** `setTimeout` в фоне продолжает работать (с троттлингом).

Так сделано везде: у колеса — сторожевой таймер, у Crash — краш по таймеру, у Слотов/Костей/бонусов — тоже таймеры. Побочная польза: пропущенный кадр не может изменить выигрыш.

### 3. Заскоупленный CSS не достаёт до элемента в `<body>`

Анимация летящей фишки была написана как `#rouletteGame .flying-chip`, а сам элемент добавлялся в `document.body` — правило не совпадало, фишка летела невидимым `div`. Плюс переменная `--serif` тоже заскоуплена и снаружи не определена.

**Правило:** если элемент живёт вне хоста игры, его правило должно быть глобальным и самодостаточным.

### 4. `position: fixed` внутри трансформированного предка

`.game-view` имеет `transform`, а трансформированный предок делает `fixed` относительным **к себе**. Фон игры с `fixed; inset:0` накрыл шапку вместе с кнопкой «Выйти» и балансом.

**Правило:** внутри игры использовать `absolute`, а не `fixed`.

### 5. Процент от элемента без высоты

Стрип барабана — `position:absolute` без высоты, а ячейки просили `height: 33.33%`. Процент от `auto` не резолвится → ячейки схлопнулись по высоте текста, все символы уехали наверх. Лечится `height: 100%` на стрипе.

### 6. Порядок ячеек в барабане

Барабан останавливается на `translateY(0)`, где видны **верхние** ячейки стрипа. Значит результат должен лежать **в начале** стрипа, а филлеры — после. Обратный порядок приводил к остановке на случайных символах, а каскад затем впрыгивал настоящей сеткой — это выглядело как рывок.

## Производительность

Мобильные GPU захлёбывались на анимации размытых слоёв. Что важно знать:

- **Анимация размытого элемента пересчитывает блюр каждый кадр.** `mix-blend-mode` вдобавок ломает кеширование результата — он убран.
- **Пока открыта игра, оболочка невидима, но продолжала анимироваться.** Теперь при открытии игры на `body` вешается `.game-open`: фоновые пятна и бегущая строка встают на паузу, скрытые экраны не рендерятся (`content-visibility: hidden`).
- **Во время раунда** хост игры получает `.spinning` — собственные фоновые орбы тоже замирают.
- Вращающийся холст колеса вынесен в отдельный слой (`will-change: transform`), чтобы не перерисовывать обод и панель.
- Не ставьте таймер на каждый тик анимации — указатель колеса когда-то создавал десятки `setTimeout` за спин; заменено на CSS-переход.

## Что не проверено

Честный список — эти вещи написаны и задеплоены, но вживую не наблюдались:

- **Бонусы колеса Cash Hunt и Пачинко.** Бонусные сектора редкие, за ~20 автоспинов не выпали. Прикрыты страховкой (при сбое раунд закроется по базовому множителю, панель не умрёт), но глазами не видены. Монетка (Coin Flip) — проверена.
- **Анимации как таковые.** Тестовая среда считает страницу скрытой (`document.hidden: true`): `rAF` и CSS-анимации заморожены, `getComputedStyle` отдаёт устаревшие значения. Проверяемо: геометрия, логика, тайминги, состояние DOM. Не проверяемо: плавность, свечения, накрутка чисел, анимация выигрыша.
- **Производительность на реальном Android.** Оптимизации сделаны по механике (что именно перестало считаться), но FPS не замерялся.

Всё это проверяется только на живом устройстве.

## Путь к реальным деньгам

Сейчас **всё** считается на клиенте. Перед любыми реальными деньгами нужно:

**1. Перенести на сервер:**
- генератор случайных чисел и определение исхода раунда;
- расчёт выплат;
- баланс и транзакции.

Клиент должен только **показывать** результат, полученный с сервера. Точки для замены на API: объект `state` (баланс, статистика, транзакции, бонусы) и функции `setBalance`, `doDeposit`, `claimDaily`, `redeemPromo`.

**2. Проверять Telegram `initData` на сервере** — HMAC с токеном бота. `initDataUnsafe` пригоден только для отображения и никогда — для того, что платит.

**3. Provably Fair** — публикация хеша сида до раунда, раскрытие после.

**4. Платежи** — провайдер (крипто/TON/Telegram Stars), с учётом custody, AML и KYC.

**5. Юридическая часть.** Серверный RNG — необходимое, но не достаточное условие. **Реальное казино требует игровой лицензии** в каждой юрисдикции, где вы работаете; платёжные провайдеры запросят её до подключения. Об этом стоит узнать **до** вложений в бэкенд, а не после.
